- **C-020 — Ne JAMAIS utiliser `obsidian_patch_content` sur `target_type: frontmatter` pour les docs Brain au canon R-055** : L'outil MCP Obsidian `obsidian_patch_content` (qui appelle l'endpoint Local REST API `PATCH` sur le frontmatter) **parse le YAML, modifie le champ ciblé, puis re-sérialise** avec son propre formatter. - **C-021 — `obsidian_patch_content target_type=heading` : passer un path complet `H1::H2::...` (séparateur `::`)** : L'outil MCP Obsidian `obsidian_patch_content` avec `target_type: heading` exige un `target` au format **path complet depuis le H1 racine du doc**, avec `::` comme séparateur de profondeur. Le matching sur heading isolé échoue avec `Error 40080: invalid-target`. **Patterns** : (i) ❌ `target: "Section A"` → invalid-target ; (ii) ❌ `target: "Section A — heading de niveau 2"` → invalid-target ; (iii) ❌ `target: "Sous-section A.1"` (H3 isolé) → invalid-target ; (iv) ❌ `target: "Section A — heading de niveau 2/Sous-section A.1"` (séparateur `/`) → invalid-target ; (v) ✅ `target: "Test doc 1"` (H1 racine seul) → OK ; (vi) ✅ `target: "Test doc 1::Section B — heading de niveau 2"` (path `::` depuis H1) → OK. **Règle pratique** : toujours préfixer par le H1 du doc et utiliser `::` comme séparateur. - **C-022 — `obsidian_patch_content target_type=block` : ne JAMAIS inclure l'anchor `^id` dans le `content` (corruption silencieuse)** : Lors d'un `patch_content` block en mode `replace`, le serveur Local REST API **conserve l'ancien anchor** `^id` du block ET **concatène le nouveau** si on l'inclut dans le `content`. Résultat après un seul patch : `Block remplacé. ^block-test-q1 ^block-test-q1`. À chaque patch suivant, l'anchor est encore dupliqué. **Conséquence** : pollution silencieuse du doc cible (visuellement laid + cassage potentiel des transclusions `![[doc#^id]]` à terme). **Règle** : ne jamais inclure l'anchor `^id` dans le `content` passé au `patch_content` block — laisser le serveur préserver l'anchor original. **Workaround à confirmer** dans une prochaine session : passer uniquement le contenu textuel, sans `^id`. **Articulation** : C-020 (anti-pattern frontmatter) + C-021 (path heading) — ensemble ces 3 règles cernent les pièges critiques de `patch_content`. **Découverte** : 02-05-2026 (Q4 sandbox MCP Obsidian, test `patch_content` block sur `^block-test-q1`).

- **C-023 — Ne PAS utiliser la transclusion `![[Doc#Section]]` ou `![[Doc^block-id]]` dans les docs canoniques LBP** : La syntaxe Obsidian `![[]]` (avec `!`) embed visuellement le contenu cible **au rendu Obsidian uniquement**. Le contenu transclu **n'est PAS résolu dans le `.md` source** (le marqueur `![[Doc#Section]]` reste tel quel) **ni indexé par `obsidian_simple_search` côté consommateur**. Conséquence : un agent (Claude, Brain Architect, script d'audit) ou un outil (`cat`, `git grep`, `Read`) qui lit le `.md` voit le marqueur brut et **ne voit pas** le contenu transclu. Pour les docs canoniques LBP (manuels de BDD, WR-RD, taxonomies, docs méta indexés, notes de concept, templates, etc.), **toujours utiliser des wikilinks simples `[[Doc]]` ou `[[Doc#Section]]` (sans `!`)** ou des citations textuelles. La transclusion `![[]]` reste OK pour les **docs de navigation/synthèse non canoniques** (MOC/Maps of Content, dashboards, pages d'accueil destinées à l'humain en lecture Obsidian). Cette règle s'applique aussi aux **blocks Dataview** (rendus à l'affichage uniquement, invisibles dans le `.md` brut). **Articulation** : R-001 (Markdown SoT) — un doc canonique doit être lisible et auditable indépendamment de l'UI Obsidian. **Découverte** : 02-05-2026 (Q1+Q2 sandbox MCP Obsidian, tests `simple_search` sur texte présent uniquement dans la transclusion → 0 match côté consommateur ; `get_file_contents` retourne le `![[]]` brut non résolu).

- **C-024 — Pattern recommandé pour les renvois inter-docs LBP : wikilinks `[[]]` + aliases frontmatter** : Pour la navigation entre docs canoniques LBP, utiliser systématiquement la syntaxe wikilinks Obsidian `[[Doc]]`, `[[Doc#Section]]`, `[[Doc|alias custom]]`, `[[Alias frontmatter]]`. Bénéfices testés et confirmés (02-05-2026, Q1+Q2 sandbox) : (i) **lisibles humain ET agent** (texte plein dans le `.md`, indexé par `simple_search` et `complex_search`) ; (ii) **résilients au rename** : Obsidian met à jour automatiquement TOUS les wikilinks (simples, avec section, avec alias custom) ET les transclusions `![[]]` quand un doc est renommé via l'UI Obsidian (F2 ou clic droit), à condition que l'option **Settings → Files & Links → « Toujours mettre à jour les liens internes »** soit activée — testé 02-05-2026 sur 5 liens / 2 docs : 5/5 mis à jour ; (iii) **aliases frontmatter** (déclarés dans `aliases: [...]`) sont indexés par `simple_search` et permettent d'écrire `[[Concept X]]` au lieu de `[[Doc canonique X]]`, et **survivent au rename** sans modification (l'alias suit le doc renommé). **Règle pratique** : tout doc canonique susceptible d'être cité sous plusieurs formes ou renommé devrait déclarer ses aliases en frontmatter. ## 5. Protocole de capture proactive des regles

A chaque fois qu'une regle, convention ou decision emerge, je prends l'initiative de proposer sa capture.

### 5.1 Declencheurs

Je propose une capture dans ces situations :

- **Succes non-trivial** : on vient de reussir une action qui a demande un choix de methode → je propose de retenir comment on a fait.
- **Decision orientante** : tu prends une decision qui va influencer des pratiques futures → je propose capture immediate.
- **Convention implicite detectee** : je constate qu'on a applique une convention sans l'avoir formalisee → je propose de la rendre explicite.
- **Anti-pattern identifie** : on decouvre qu'une approche ne marche pas → je capture la regle negative (ce qu'il ne faut pas faire).
- **Conflit / contradiction** : deux regles existantes se marchent dessus → je signale.

### 5.2 Format de proposition

Je te presente :
- **Enonce propose** de la regle (titre + 2-3 lignes)
- **Portee** pressentie (Brain / Twin / Transverse / Contextuel a notre collab)
- **Nature** : "Regle fixe", "Hypothese a confirmer", ou "Decision archi"

### 5.3 Routage selon ton arbitrage

| Ton arbitrage | Destination |
|---|---|
| "Regle fixe" intrinseque a l'ecosysteme | `refs/RULES_LBP.md` avec ID R-XXX |
| "Hypothese a confirmer" | `refs/RULES_BRAIN_TWIN-backlog.md` |
| "Contextuel a notre collab" (comportement Claude, outillage) | `CLAUDE.md` avec ID C-XXX |
| "Decision archi" (choix structurant) | `refs/DECISIONS_LBP.md` avec ID D-XXX |
| "Laisser pour plus tard" | rien, on avance |

### 5.4 Apres capture

Je suggere un commit. Je mets a jour `refs/ECOSYSTEM-STATE.md` si l'etat a change.
